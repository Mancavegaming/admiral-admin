# AdmiralsPanel v0.8.0 — Server-side item spawning

This release ships a **Spawn** tab that finally puts items into players' chests *for real*. Previous give-item commands (v0.4–v0.5) appeared to add items to the HUD but they vanished on the next save — the auto-pickup pipeline never committed server-side. v0.8 takes a different approach: write items directly into the world RocksDB save, with a one-click installer that handles all the plumbing.

## Highlights

- 🎯 **Spawn tab in the panel UI** — searchable autocomplete of all 1268 items, full asset-path resolution, live progress display, admin-only auth.
- 🔌 **One-command install** — the updated `install.ps1` detects Python, installs deps, deploys the spawn tool, and registers a Scheduled Task for the sidecar. Total: one PowerShell command from a fresh server.
- 🔐 **Persistent sessions** — admin login survives server restarts now (was wiped on every restart in v0.7).
- 🛠️ **Production-ready architecture** — sidecar pattern with single-instance lock, status polling, and graceful failure recovery.

## What's new

### Spawn tab

Pick a target chest (by 32-char `BuildingBlock` key OR by marker item placed in slot 0), pick from a dropdown of all 1268 known items, set count, click Spawn. The server briefly restarts (~10s) and the item appears in the chest's first empty slot.

The UI walks you through `queued → stopping_server → writing_db → restarting_server → done`. It tolerates the brief panel-down window during the restart and resumes polling automatically. If players are connected, you get a confirmation dialog before the spawn fires.

### Item catalog endpoint

New `ap.itemcatalog` native command returns JSON `{count, items: [{name, path}, ...]}` for every loaded `UR5BLInventoryItem`. The Spawn tab uses this to populate the autocomplete dropdown so admins can find items by name even if they're not in any current chest.

### Sidecar architecture

The panel runs inside the WindroseServer process, so it can't orchestrate its own restart. v0.8 introduces an out-of-process Python sidecar (`tools/spawn-item/sidecar.py`) that:

- Watches `admiralspanel_data/rcon/spawn/` for new request files
- Stops the server, opens RocksDB with `compression=none` (Windrose's build has Snappy disabled)
- Decodes the target chest's BSON, inserts an `FR5BLItem` with a fresh UUID into the first empty slot, re-encodes
- Writes back, restarts the server, polls until the panel responds
- Writes status updates the UI polls

Single-instance enforced via `msvcrt.locking`. Auto-started by Scheduled Task at boot + logon.

### Standalone CLI

For terminal users or scripted workflows, `tools/spawn-item/spawn_item.py` provides the same logic as a one-shot CLI:

```powershell
python spawn_item.py --chest <32-hex> --item-substr Wood_T01 --count 999
python spawn_item.py --marker ScallopShell --item-path /R5BusinessRules/.../DA_X.DA_X --count 50
```

## For new admins — install

```powershell
git clone https://github.com/Mancavegaming/admiral-admin.git
cd admiral-admin
powershell -ExecutionPolicy Bypass -File install.ps1
```

The installer (5 steps):
1. Deploy Lua mod
2. Deploy web UI
3. Deploy native DLL
4. **(new)** Detect Python 3.11+, `pip install rocksdict pymongo`, copy spawn-item tool, register Scheduled Task `AdmiralsPanel-Spawn-Sidecar`
5. Print next steps + the auto-generated random password location

After the first server start, open `http://localhost:8790/` and log in with the password from `admiralspanel.json`. Each install gets its own unique 24-char alphanumeric password.

Pass `-SkipSidecar` to opt out of step 4 if you'd rather manage the sidecar yourself.

## For upgraders from v0.7

```powershell
cd admiral-admin
git pull
powershell -ExecutionPolicy Bypass -File install.ps1
```

The `main.dll` swap requires the server to be stopped (the installer detects this and warns). Everything else can deploy while the server is running. Multiplier values in `admiralspanel.json` are preserved.

If you used `ap.giveitem` from v0.4–v0.5: it's still present but flagged as broken in docs. Use the Spawn tab instead.

## Requirements

- Windows Server 2019/2022/2025 (or Win10/11 desktop)
- Windrose Dedicated Server (Steam DS SKU)
- UE4SS at `R5/Binaries/Win64/ue4ss/`
- **Python 3.11+** with "Add to PATH" (new in v0.8 — required only if you want the Spawn feature; pass `-SkipSidecar` to skip)
- PowerShell 5.1+ (ships with Windows Server)

## Known limitations

- **Spawn requires server restart (~10s downtime)** — RocksDB needs an exclusive lock for writes. Connected players get disconnected. The UI warns and asks for confirmation.
- **Each chest has 16 slots** — if all are filled, the spawn errors out. Empty a slot first.
- **Item must already exist as a loaded `UR5BLInventoryItem`** — if Windrose's pak files don't contain the asset, neither the dropdown nor a manual path will work. Standard items all work.

## Architecture notes

For the curious: we proved server-side memory writes don't replicate to clients (the engine uses UPROPERTY/RPC channels, not raw memory), and that the v0.4 teleport-based give-item never authoritatively committed to the inventory hierarchy. The save-file write is the only path that produces real, persistent items. Full RE notes in [`tools/spawn-item/README.md`](../../tools/spawn-item/README.md).
